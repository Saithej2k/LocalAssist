# LocalAssist CLI Baseline - 2026-07-10

Command:

```bash
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer swift run -c release localassist-bench --iterations 100 --warmup 5 --concurrency 4 --json --output docs/performance/2026-07-10-benchmark.json
```

This benchmark measures the deterministic offline fallback path for repeatable
local and CI verification. It is separate from the Xcode Instruments Foundation
Models profile and from the live model numbers in `docs/evals`.

| Metric | Value |
| --- | ---: |
| Success count | 100 |
| Failure count | 0 |
| Fallback rate | 1.00 |
| p50 latency | 20.421 ms |
| p95 latency | 20.888 ms |
| p99 latency | 21.102 ms |
| Standard deviation | 0.332 ms |
| Peak memory | 12.234 MB |
| Peak memory delta | 0.656 MB |
| Throughput | 194.326 runs/sec |
| Cancellation succeeded | true |
| Cancellation latency | 0.028 ms |

The p95 delta against 2026-07-06 (155.6 ms → 20.9 ms, roughly 7×) is an engine
change, not machine variance: since that baseline the due-date parser started
caching its `DateFormatter`s (previously up to fifty-six allocations per parse
attempt), `RunHistoryStore` gained an in-memory cache so history mutations stop
re-decoding fifty runs of JSON from disk, and the transcript chunker moved to
`NLTokenizer`. The same audit round that made those changes recorded the test
suite dropping from about three seconds to under one; this baseline is the
benchmark catching up with it. The tight standard deviation (0.33 ms across
100 runs) says the pipeline is now allocation-quiet enough that the remaining
cost is the work itself.
